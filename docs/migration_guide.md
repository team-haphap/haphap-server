# DB 마이그레이션 규칙 (Flyway)

이 프로젝트는 DB 스키마를 Flyway로 관리합니다. 스키마 변경은 반드시 이 규칙을 따릅니다.

## 핵심 원칙

DB 구조를 바꾸려면 **손으로 마이그레이션 SQL 파일을 작성**합니다. 손으로 DB에 직접 `ALTER TABLE`을 치거나, 엔티티만 바꿔서 스키마가 자동으로 바뀌길 기대하지 않습니다. (`ddl-auto`는 `validate`로 고정되어 있어 Hibernate는 스키마를 변경하지 않고 검증만 합니다.)

## 언제 마이그레이션 파일이 필요한가

**필요함 — DB 구조가 바뀌는 변경**
- 필드 추가/삭제 (컬럼 추가/삭제)
- 필드 타입·길이 변경
- 새 엔티티 클래스 (새 테이블)
- `@Column(nullable)`, `@UniqueConstraint`, `@Index` 등 제약·인덱스 변경
- 연관관계(`@ManyToOne` 등) 추가로 FK 컬럼이 생기는 변경

**불필요함 — DB와 무관한 변경**
- 메서드 추가/수정 (비즈니스 로직)
- 생성자, 정적 팩토리, `@Getter` 등
- 필드 매핑은 그대로 두고 하는 리팩터링, 주석 정리

## 작성 방법

1. `src/main/resources/db/migration/` 에 파일 생성
2. 파일명 규칙: `V{다음번호}__{설명}.sql`
    - 예: `V2__drop_registration_stage.sql`, `V3__add_nickname_to_users.sql`
    - `V` 대문자, 언더스코어 2개(`__`), 번호는 항상 기존 최대보다 크게
3. 파일 안에 변경 SQL 작성
   ```sql
   ALTER TABLE users ADD COLUMN nickname VARCHAR(50);
   ```
4. 필요하면 엔티티도 함께 수정 (SQL과 일치하도록)
5. 커밋 → 앱 실행 시 Flyway가 자동 적용

## 절대 하지 말 것

- **이미 병합된 마이그레이션 파일 수정 금지.** Flyway가 체크섬으로 검증하므로 수정하면 에러가 납니다. 잘못은 새 파일(다음 번호)로 바로잡습니다.
- **번호 역행·중복 금지.** 항상 현재 최대 번호보다 큰 번호를 씁니다. 브랜치 작업 중 번호가 겹치면 병합 전에 조율합니다.
- **민감 정보 커밋 금지.** 마이그레이션 SQL(구조)은 커밋하지만, 비밀번호·시크릿·키 파일은 환경변수/별도 관리로 분리합니다.
